# FAQ

## General

### What is Stroomprijsprognose?

A Dutch website ([stroomprijsprognose.nl](https://stroomprijsprognose.nl)) that provides hourly electricity price forecasts for several European countries. It combines EPEX SPOT day-ahead auction prices with machine learning forecasts.

### What do the prices include?

Retail prices include:
- Wholesale electricity price (day-ahead or forecast)
- Grid fees (Netzentgelte) for your local grid operator
- Supplier markup
- Levies and taxes (excluding VAT)
- VAT (Mehrwertsteuer)

The exact components depend on your postal code.

### Is this integration free?

Yes. Both the API and this integration are free. No API key required.

### How accurate are the forecasts?

The API itself reports accuracy metrics. General expectations:
- Day-ahead prices: exact (these are auction results)
- Model forecasts (12+ hours out): directionally useful, ±2-5 ct/kWh typical error
- Longer horizons: wider error margins

Do not make financial decisions solely based on forecasts.

## Technical

### How often does the data update?

Default: every 15 minutes. Configurable between 5–60 minutes in options.

The underlying API data updates approximately every 15 minutes. Polling more often than that provides no benefit.

### Does it work offline?

No. The integration requires internet access to reach `stroomprijsprognose.nl`. If the API is unreachable, the last fetched data persists until the next successful fetch.

### What happens when the API is down?

The coordinator retries automatically. Sensors retain their last known values. After several failures, sensors show `unavailable`.

### Can I use multiple postal codes?

Yes. Add the integration multiple times with different postal codes. Each instance creates independent sensors.

### How do I override the supplier markup?

The integration stores overrides in options but does not yet recalculate sensor values client-side. To apply custom markups today, use a template sensor:

```jinja2
{% set base = state_attr('sensor.stroomprijsprognose_current_price', 'hourly_prices')[0].retail_total_ct_kwh %}
{% set custom_markup = 1.5 %}
{% set custom_vat = 19 %}
{{ ((base / 1.19) + custom_markup) * 1.19 | round(2) }}
```

Client-side recalculation is planned for a future release.

### Why are some sensors disabled by default?

Sensors like `current_forecast_price` and `price_source` are for advanced users. Enable them in the entity registry if needed.

## Development Specific

### How was this built?

The initial codebase was generated by AI (Claude Code) and has been iteratively reviewed and improved. 58 automated tests cover coordinator logic, sensor values, config flow validation, and caching behavior.

### Should I trust this in production?

For non-critical automations (scheduling appliances, notifications): yes, the risk is low. For financial or safety-critical use: validate the data against the API directly first.

### Will this be maintained?

Depends. Bug reports and PRs are welcome.

### How was it developed?

1. User provided the API URL
2. AI fetched the API to understand the data structure
3. AI designed the architecture (standard HA coordinator pattern)
4. AI generated all 10 files simultaneously in one batch
5. AI reviewed and fixed 2 bugs (`forecast_slots` unit, `vol.Optional` `description` params)
6. HACS discovered — 3 more fixes applied (`integration_type`, `@callback`, `services.yaml`)
7. Tests + CI/CD added (58 tests, mock HA package)
8. Iterative improvements: robust timestamp parsing, smart API caching, service schema validation

### Are there tests?

Yes. 58 pytest tests covering coordinator data processing, sensor values, config flow schema, caching behavior, and unit handling. Run with `pip install -r requirements-test.txt && pytest tests/ -v`.

### How do I create a release?

Use the GitHub Actions workflow: **Actions → Create Release → Run workflow**, enter version tag like `v1.0.0`. The workflow builds a zip with version in manifest.json, creates a git tag, and publishes a GitHub release.

### I found a bug. Now what?

Open an issue. Better yet, open a PR — the original author is an AI and won't fix it without prompting.
